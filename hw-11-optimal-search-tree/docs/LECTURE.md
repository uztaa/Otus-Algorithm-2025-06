# Лекция: Деревья Оптимального Поиска (ДОП)

## 1. Введение

### 1.1 Мотивация

До сих пор мы предполагали, что **частота обращения ко всем ключам одинакова**. Однако в реальных системах это редкость:

| Сценарий | Особенность |
|----------|-------------|
| Сканер транслятора | Служебные слова (`if`, `while`) встречаются чаще |
| Кэш веб-сервера | Популярные страницы запрашиваются в 1000× чаще |
| Словарь | Частотные слова (`the`, `a`, `is`) доминируют |

**Закон Ципфа:** В естественных данных частота элемента обратно пропорциональна его рангу:
```
частота(r) ∝ 1/r
```

### 1.2 Идея оптимизации

Если мы **знаем частоты** обращения к ключам заранее, можно построить дерево так, чтобы:
- Частые ключи были **ближе к корню**
- Редкие ключи — дальше от корня

Это уменьшит **среднее время поиска**.

---

## 2. Формальное определение

### 2.1 Средневзвешенная высота

Пусть дерево содержит n вершин V₁, V₂, ..., Vₙ с весами w₁, w₂, ..., wₙ.

**Определения:**
- `W = Σwᵢ` — суммарный вес
- `hᵢ` — глубина вершины Vᵢ (корень на глубине 1)

**Средневзвешенная высота:**
```
hср = (w₁·h₁ + w₂·h₂ + ... + wₙ·hₙ) / W = Σ(wᵢ·hᵢ) / W
```

### 2.2 Дерево оптимального поиска

**Дерево оптимального поиска (ДОП)** — дерево поиска с **минимальной** средневзвешенной высотой для заданного набора ключей и весов.

### 2.3 Пример

Рассмотрим ключи V₁=1, V₂=2, V₃=3 с весами w₁=60, w₂=30, w₃=10 (W=100).

**Вариант A: Корень = 1**
```
    1 (h=1)
     \
      2 (h=2)
       \
        3 (h=3)

hср = (60·1 + 30·2 + 10·3) / 100 = 150/100 = 1.5 ✓ ОПТИМАЛЬНО
```

**Вариант B: Корень = 2**
```
      2 (h=1)
     / \
    1   3
  (h=2) (h=2)

hср = (60·2 + 30·1 + 10·2) / 100 = 170/100 = 1.7
```

**Вариант C: Корень = 3**
```
        3 (h=1)
       /
      2 (h=2)
     /
    1 (h=3)

hср = (60·3 + 30·2 + 10·1) / 100 = 250/100 = 2.5 ✗ ХУДШИЙ
```

**Вывод:** Ключ с максимальным весом (60) должен быть в корне → hср минимальна.

## 3. Точный алгоритм (Кнут)

### 3.1 Динамическое программирование

Оптимальное ДОП можно построить точно за **O(n³)** с помощью ДП.

Пусть `C[i][j]` — минимальная стоимость поддерева для ключей i..j.

**Рекуррентное соотношение:**
```
C[i][j] = min(C[i][k-1] + C[k+1][j]) + W[i][j], для всех k ∈ [i,j]
```

где `W[i][j] = Σw[i..j]` — суммарный вес сегмента.

### 3.2 Недостатки точного алгоритма

| Характеристика | Значение |
|----------------|----------|
| Время | O(n³) |
| Память | O(n²) |
| Практичность | Только для малых n (< 1000) |

**Вывод:** Для больших данных нужны **приближённые** алгоритмы.

## 4. Приближённый алгоритм 1: Сортировка по весам

### 4.1 Идея

Вставлять ключи в обычное BST в порядке **убывания весов**.

Интуиция: ключи с большим весом вставляются первыми → оказываются ближе к корню.

### 4.2 Алгоритм

```
Алгоритм1(keys, weights):
    pairs = zip(keys, weights)
    sort(pairs, by=weight, descending)
    
    tree = новое BST
    for (key, weight) in pairs:
        tree.insert(key)
    
    return tree
```

### 4.3 Пример

Ключи: А, В, Е, И, К, Л, Н, О, П, Р, Т, У
Веса:  4, 3, 2, 1, 2, 1, 2, 3, 1, 2, 1, 1

**После сортировки по весам:**
```
А(4), В(3), О(3), Е(2), К(2), Н(2), Р(2), И(1), Л(1), П(1), Т(1), У(1)
```

**Построение BST:**
1. Вставляем А → корень
2. Вставляем В → правее А
3. Вставляем О → правее В
4. и т.д.

**Результат:** hср ≈ 3.39

### 4.4 Анализ

| Характеристика | Значение |
|----------------|----------|
| Сложность | O(n log n) сортировка + O(n²) вставка в BST |
| Качество | **Плохое** — результат близок к случайному дереву |
| Проблема | Порядок вставки определяется весами, а не ключами |

**Почему плохой?** При вставке в BST структура зависит от **порядка** вставки, а не только от весов. Ключ с большим весом может оказаться глубоко, если его ключевое значение "неудобное".

## 5. Приближённый алгоритм 2: Центр тяжести

### 5.1 Идея

Выбирать **корнем** узел, который минимизирует разность весов левого и правого поддеревьев.

Аналогия: найти "центр масс" сегмента и поставить его в корень.

### 5.2 Алгоритм

```
Алгоритм2(keys, weights):
    sort(keys)  // по значению ключа!
    return buildRecursive(keys, weights, 0, n-1)

buildRecursive(keys, weights, left, right):
    if left > right:
        return null
    
    k = findCenterOfGravity(weights, left, right)
    
    node = new Node(keys[k], weights[k])
    node.left = buildRecursive(keys, weights, left, k-1)
    node.right = buildRecursive(keys, weights, k+1, right)
    
    return node

findCenterOfGravity(weights, left, right):
    // Найти k, где |leftWeight - rightWeight| минимально
    // leftWeight = Σw[left..k-1]
    // rightWeight = Σw[k+1..right]
    
    totalWeight = sum(weights[left..right])
    
    bestK = left
    minDiff = |totalWeight - 2·weights[left]|
    
    for k = left to right:
        leftWeight = sum(weights[left..k-1])
        rightWeight = sum(weights[k+1..right])
        diff = |leftWeight - rightWeight|
        
        if diff < minDiff:
            minDiff = diff
            bestK = k
    
    return bestK
```

### 5.3 Оптимизация с префиксными суммами

Вычисление `sum(weights[i..j])` за O(1):

```
prefixSum[0] = 0
prefixSum[i] = prefixSum[i-1] + weights[i-1]

sum(weights[left..right]) = prefixSum[right+1] - prefixSum[left]
```

### 5.4 Пример

Те же ключи, отсортированные по алфавиту:

| А | В | Е | И | К | Л | Н | О | П | Р | Т | У |
|---|---|---|---|---|---|---|---|---|---|---|---|
| 4 | 3 | 2 | 1 | 2 | 1 | 2 | 3 | 1 | 2 | 1 | 1 |

**Суммарный вес:** W = 23

**Шаг 1: Найти центр тяжести**
- Ищем k, где leftWeight ≈ rightWeight
- k=5 (К): leftWeight=10, rightWeight=10 → разность=0 ✓

**Корень = К**

**Шаг 2: Рекурсия для поддеревьев**
- Левое: [А, В, Е, И] с весами [4, 3, 2, 1]
- Правое: [Л, Н, О, П, Р, Т, У] с весами [1, 2, 3, 1, 2, 1, 1]

**Результат:** hср ≈ 2.82

### 5.5 Анализ

| Характеристика | Значение |
|----------------|----------|
| Сложность | O(n log n) |
| Качество | **Хорошее** — асимптотически оптимально |
| Преимущество | Структура определяется балансом весов |

## 6. Сравнение алгоритмов

### 6.1 Теоретическое сравнение

| Критерий | Алгоритм 1 | Алгоритм 2 |
|----------|------------|------------|
| Сложность | O(n²) | O(n log n) |
| Качество hср | ~случайное | ~оптимальное |
| Сходимость | Плохая | Асимптотическая |
| Простота | Простой | Средняя |

### 6.2 Практические результаты (из примера)

| Алгоритм | hср |
|----------|-----|
| Алгоритм 1 | 3.39 |
| Алгоритм 2 | 2.82 |
| Улучшение | **17%** |

### 6.3 Интересное наблюдение: Zipf-распределение

При **Zipf-распределении** весов (когда несколько ключей имеют очень высокие веса, а остальные — низкие) Алгоритм 1 может показать результаты **лучше** Алгоритма 2!

**Почему?** При сортировке по весам ключи с максимальными весами вставляются первыми и автоматически оказываются близко к корню. Это идеально соответствует Zipf-распределению.

**Бенчмарк (N=10000, Zipf):**
| Алгоритм | hср | Поиск (Zipf) |
|----------|-----|--------------|
| Алгоритм 1 | 8.11 | 19.9 ms |
| Алгоритм 2 | 8.61 | 21.5 ms |
| AVL | - | 29.2 ms |

**Вывод:** Теоретически "плохой" Algo1 может быть эффективен для специфических распределений!

## 7. Применение ДОП

### 7.1 Когда использовать ДОП?

✅ **Подходит:**
- Статический набор ключей (не меняется)
- Известны частоты/веса заранее
- Требуется максимальная скорость поиска

❌ **Не подходит:**
- Динамические данные (частые вставки/удаления)
- Неизвестное распределение запросов
- Равновероятные обращения

### 7.2 Примеры применения

| Область | Пример |
|---------|--------|
| Компиляторы | Таблица зарезервированных слов |
| Сетевые протоколы | Таблица маршрутизации |
| Базы данных | Статические справочники |
| Сжатие данных | Кодовые таблицы (Хаффман) |

## 8. Связь с другими структурами

### 8.1 Splay-дерево (расширяющееся)

Splay-дерево **адаптивно** перестраивает структуру на основе запросов:
- Часто запрашиваемые ключи "всплывают" к корню
- Не требует знания весов заранее

**Преимущество:** Работает для динамических данных
**Недостаток:** Амортизированная сложность, накладные расходы на перестройку

### 8.2 Декартово дерево (Treap) как ДОП

Если использовать **веса как приоритеты** в Treap:
- Ключ с максимальным весом → максимальный приоритет → корень
- Структура автоматически оптимизируется

```cpp
// Treap с весами как приоритетами
for (int i = 0; i < n; ++i) {
    treap.insert(keys[i], weights[i]);  // weight как priority
}
```

**Это эквивалентно ДОП!**

## 9. Реализация в коде

### 9.1 Базовый класс

```cpp
class OptimalTreeBase : public IBinaryTree {
public:
    // Запрещаем динамические операции
    void insert(int x) final override {
        throw std::logic_error("Use build() for optimal trees");
    }
    
    void remove(int x) final override {
        throw std::logic_error("Optimal tree is static");
    }
    
    // API для ДОП
    virtual void build(const std::vector<int>& keys,
                       const std::vector<int>& weights) = 0;
    
    double getWeightedHeight() const;  // hср
};
```

### 9.2 Алгоритм 2 (ключевая функция)

```cpp
int findCenterOfGravity(int left, int right, 
                        const std::vector<int64_t>& prefixSum) {
    int bestIdx = left;
    int64_t minDiff = INT64_MAX;
    
    for (int k = left; k <= right; ++k) {
        int64_t leftWeight = prefixSum[k] - prefixSum[left];
        int64_t rightWeight = prefixSum[right + 1] - prefixSum[k + 1];
        int64_t diff = std::abs(leftWeight - rightWeight);
        
        if (diff < minDiff) {
            minDiff = diff;
            bestIdx = k;
        }
    }
    
    return bestIdx;
}
```

## 10. Итоги

| Концепция | Описание |
|-----------|----------|
| **ДОП** | Дерево с минимальной средневзвешенной высотой |
| **hср** | Σ(wᵢ·hᵢ) / W — метрика качества |
| **Алгоритм 1** | Сортировка по весам → BST (плохой) |
| **Алгоритм 2** | Центр тяжести (хороший, O(n log n)) |
| **Применение** | Статические данные с известным распределением |

### Ключевые выводы

1. **ДОП эффективно**, когда распределение запросов известно и стабильно
2. **Алгоритм 2** — предпочтительный приближённый метод
3. **Для динамических данных** лучше Splay или Treap с адаптацией
4. **Treap с весами** — элегантная альтернатива построению ДОП
