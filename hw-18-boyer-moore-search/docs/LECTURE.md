# Алгоритм Бойера-Мура: Поиск подстроки в тексте

## 1. Введение

### 1.1 Постановка задачи

**Задача поиска подстроки** (pattern matching) — это фундаментальная задача информатики, которая заключается в нахождении всех вхождений строки-образца (pattern) в тексте (text).

**Входные данные:**
- `text` — исходный текст длины `n`
- `pattern` — искомый образец (шаблон) длины `m`

**Выходные данные:**
- Позиция первого вхождения образца в тексте (или -1, если не найден)
- Либо список всех позиций вхождений

### 1.2 Области применения

- Текстовые редакторы (команды "Найти" и "Заменить")
- Поисковые системы
- Биоинформатика (поиск последовательностей ДНК)
- Антивирусное ПО (поиск сигнатур)
- Системы обнаружения вторжений (IDS)
- Компиляторы (лексический анализ)

### 1.3 Обзор алгоритмов

| Алгоритм | Препроцессинг | Худший случай | Лучший случай | Год |
|----------|---------------|---------------|---------------|-----|
| Наивный | O(1) | O(n·m) | O(n) | - |
| KMP | O(m) | O(n+m) | O(n) | 1977 |
| Boyer-Moore-Horspool | O(m+σ) | O(n·m) | O(n/m) | 1980 |
| Boyer-Moore | O(m+σ) | O(n+m) | O(n/m) | 1977 |

где `σ` — размер алфавита.

---

## 2. Алгоритм полного перебора (Naive Algorithm)

### 2.1 Идея алгоритма

Самый простой подход — последовательно проверять все возможные позиции образца в тексте, сравнивая символы слева направо.

### 2.2 Визуализация

```
Текст:    K O L K O K O L O K O L L
Шаблон:   K O L O K O L
          ↑ ↑ ↑ ✗         (несовпадение на позиции 3: 'K' ≠ 'O')
          
Сдвиг на 1 позицию:
Текст:    K O L K O K O L O K O L L
            K O L O K O L
            ✗             (несовпадение на позиции 0)
            
И так далее...
```

### 2.3 Псевдокод

```
NAIVE-SEARCH(text, pattern):
    n = length(text)
    m = length(pattern)
    
    for t = 0 to n - m:
        j = 0
        while j < m AND text[t + j] == pattern[j]:
            j = j + 1
        
        if j == m:
            return t    // Найдено совпадение на позиции t
    
    return -1           // Образец не найден
```

### 2.4 Анализ сложности

- **Временная сложность:**
  - Лучший случай: O(n) — первый символ образца не совпадает
  - Худший случай: O(n·m) — например, `text = "aaaa...a"`, `pattern = "aaab"`
  - Средний случай: O(n)

- **Пространственная сложность:** O(1)

### 2.5 Недостатки

1. При несовпадении сдвигается только на 1 позицию
2. Не использует информацию о уже сравненных символах
3. Может делать много лишних сравнений

---

## 3. Алгоритм обратного сравнения (Backward Matching)

### 3.1 Идея

Единственное отличие от наивного алгоритма — сравнение символов **справа налево** (от конца образца к началу). Это создаёт основу для оптимизаций Бойера-Мура.

### 3.2 Визуализация

```
Текст:    K O L K O K O L O K O L L
Шаблон:   K O L O K O L
                      ↑   (начинаем сравнение с последнего символа)
          
Сравниваем: pattern[6]='L' с text[6]='O' → несовпадение
```

### 3.3 Преимущество обратного сравнения

При несовпадении мы уже знаем:
1. Какой символ текста не совпал (стоп-символ, bad character)
2. Какие символы справа совпали (совпавший суффикс, good suffix)

Эта информация позволяет делать **умные сдвиги** — больше чем на 1 позицию!

## 4. Алгоритм Бойера-Мура-Хорспула (BMH)

### 4.1 История

Алгоритм предложен **Найджелом Хорспулом** в 1980 году как упрощённая версия алгоритма Бойера-Мура. Использует только **правило плохого символа** (bad character rule).

### 4.2 Основная идея

При несовпадении смотрим на символ текста, который стоит **напротив последнего символа образца**, и сдвигаем образец так, чтобы этот символ совпал с его ближайшим вхождением в образце (кроме последней позиции).

### 4.3 Таблица сдвигов (Bad Character Table)

Для каждого символа алфавита вычисляем сдвиг:
- Если символ есть в образце (кроме последней позиции): `shift = m - 1 - последняя_позиция_в_образце`
- Если символа нет в образце: `shift = m` (длина образца)

**Пример для образца `KOLOKOL` (m=7):**

```
Позиции:  0 1 2 3 4 5 6
Образец:  K O L O K O L

Символ | Последняя позиция (кроме pos 6) | Сдвиг
-------|----------------------------------|------
K      | 4                                | 7-1-4 = 2
O      | 5                                | 7-1-5 = 1
L      | 2                                | 7-1-2 = 4
Другие | -                                | 7
```

### 4.4 Псевдокод

```
BMH-PREPROCESS(pattern):
    m = length(pattern)
    shift[0..255] = m              // Для всех символов ASCII
    
    for i = 0 to m - 2:            // Не включая последний символ!
        shift[pattern[i]] = m - 1 - i
    
    return shift

BMH-SEARCH(text, pattern):
    n = length(text)
    m = length(pattern)
    shift = BMH-PREPROCESS(pattern)
    
    t = 0
    while t <= n - m:
        j = m - 1                   // Начинаем с конца образца
        
        while j >= 0 AND text[t + j] == pattern[j]:
            j = j - 1
        
        if j < 0:
            return t                // Найдено!
        
        t = t + shift[text[t + m - 1]]  // Сдвиг по последнему символу
    
    return -1
```

### 4.5 Визуализация работы

```
Текст:    K O L K O K K K O L O K O L L
Шаблон:   K O L O K O L
                      ↑   text[6]='K', shift['K']=2
                      
Сдвиг на 2:
Текст:    K O L K O K K K O L O K O L L
              K O L O K O L
                          ↑   text[8]='K', shift['K']=2

Сдвиг на 2:
Текст:    K O L K O K K K O L O K O L L
                  K O L O K O L
                              ↑   и так далее...
```

### 4.6 Анализ сложности

- **Препроцессинг:** O(m + σ), где σ — размер алфавита
- **Поиск:**
  - Лучший случай: **O(n/m)** — субли­нейная сложность!
  - Худший случай: O(n·m) — например, `text = "aaa...a"`, `pattern = "ba"`
  - Средний случай: O(n)

### 4.7 Достоинства и недостатки

**Достоинства:**
- Простота реализации
- Отличная производительность на практике
- Субли­нейная сложность в лучшем случае

**Недостатки:**
- Квадратичная сложность в худшем случае
- Не использует информацию о совпавшем суффиксе

## 5. Алгоритм Бойера-Мура (Full Boyer-Moore)

### 5.1 История

Алгоритм разработан **Робертом Бойером** и **Джеем Муром** в 1977 году. Считается одним из наиболее эффективных алгоритмов поиска подстроки для практических применений.

> "Алгоритм Бойера-Мура считается наиболее быстрым среди алгоритмов общего назначения, предназначенных для поиска подстроки в строке."

### 5.2 Ключевые концепции

Алгоритм использует **два правила сдвига** и выбирает максимальный из них:

1. **Bad Character Rule** (правило плохого символа) — аналогично BMH
2. **Good Suffix Rule** (правило хорошего суффикса) — использует информацию о совпавшей части

```
shift = max(bad_character_shift, good_suffix_shift)
```

### 5.3 Терминология

- **Стоп-символ (Bad Character)** — символ текста, который НЕ совпал с соответствующим символом образца
- **Совпавший суффикс (Good Suffix)** — часть образца, которая совпала с текстом (справа от точки несовпадения)
- **Префикс** — начальная подстрока образца
- **Суффикс** — конечная подстрока образца

### 5.4 Правило плохого символа (Bad Character Rule)

При несовпадении на позиции `i` образца с символом `c` текста:

1. Если `c` есть в образце **левее позиции `i`**: сдвигаем образец так, чтобы ближайшее вхождение `c` (слева от `i`) совпало с позицией в тексте
2. Если `c` нет в образце или есть только правее: сдвигаем образец **за** этот символ

```
Текст:    ... X Y Z A B C D ...
Образец:      A B C E B C
              ↑ ↑ ↑ ✗       (несовпадение: 'A' ≠ 'E')
              
'A' есть в образце на позиции 0.
Сдвигаем так, чтобы 'A' в образце встал под 'A' в тексте.
```

### 5.5 Правило хорошего суффикса (Good Suffix Rule)

Это более сложное правило. При несовпадении, если часть образца `t` уже совпала (суффикс):

**Случай 1:** Такой же суффикс `t` встречается **ещё раз** в образце, причём символ перед ним **другой**
→ Сдвигаем образец так, чтобы это вхождение `t` совпало с текстом

**Случай 2:** Суффикс `t` не встречается больше в образце, но **часть суффикса** является **префиксом** образца
→ Сдвигаем образец так, чтобы этот префикс совпал с концом `t` в тексте

**Случай 3:** Ни одно из условий не выполнено
→ Сдвигаем образец на всю его длину

### 5.6 Визуализация правила хорошего суффикса

```
Образец: K O L O K O L
         0 1 2 3 4 5 6

Суффикс "L" (1 символ совпал):
  - "L" встречается на позиции 2
  - Перед ней "O" (позиция 1), а перед позицией 6 тоже "O"
  - Символы одинаковые! Ищем дальше или используем префикс
  - Префикс "KOL" не совпадает с "L"
  - Сдвиг = 4

Суффикс "OL" (2 символа совпали):
  - "OL" встречается на позиции 1-2
  - Перед ней "K" (позиция 0), а перед позицией 5 "K"
  - Опять одинаковые! Сдвиг = 4

И так далее...
```

### 5.7 Таблица суффиксов для `KOLOKOL`

| Совпавший суффикс | Длина | Сдвиг | Пояснение |
|-------------------|-------|-------|-----------|
| (ничего) | 0 | 1 | Минимальный сдвиг |
| L | 1 | 4 | Нет другого "L" с другим предшествующим символом |
| OL | 2 | 4 | "OL" на позиции 1-2, но предшествует тот же символ |
| KOL | 3 | 4 | "KOL" есть в начале как префикс |
| OKOL | 4 | 4 | Суффикс "KOL" = префикс |
| LOKOL | 5 | 4 | Суффикс "KOL" = префикс |
| OLOKOL | 6 | 4 | Суффикс "KOL" = префикс |

### 5.8 Псевдокод построения таблицы суффиксов

```
BUILD-SUFFIX-TABLE(pattern):
    m = length(pattern)
    suffix[0..m-1] = 0
    suffix[0] = 1                    // При 0 совпавших — сдвиг на 1
    
    for i = 1 to m - 1:              // i — длина совпавшего суффикса
        for k = 1 to m:              // k — потенциальный сдвиг
            // Проверяем, можно ли сдвинуть на k позиций
            match = true
            
            // Проверка совпадения суффиксов
            for j = 0 to i - 1:
                if (j + k + 1) > m:
                    break            // Вышли за границы
                if pattern[m - 1 - j] != pattern[m - 1 - j - k]:
                    match = false
                    break
            
            // Проверка, что предшествующий символ другой
            // или мы вышли за границу образца
            if match:
                if (m - 1 - i - k) < 0 OR pattern[m - 1 - i] != pattern[m - 1 - i - k]:
                    suffix[i] = k
                    break
    
    return suffix
```

### 5.9 Полный псевдокод алгоритма Бойера-Мура

```
BOYER-MOORE-SEARCH(text, pattern):
    n = length(text)
    m = length(pattern)
    
    // Препроцессинг
    badChar = BUILD-BAD-CHARACTER-TABLE(pattern)
    goodSuffix = BUILD-SUFFIX-TABLE(pattern)
    
    t = 0
    while t <= n - m:
        j = m - 1
        
        // Сравнение справа налево
        while j >= 0 AND text[t + j] == pattern[j]:
            j = j - 1
        
        if j < 0:
            return t            // Найдено совпадение!
        
        // Вычисление сдвигов
        matchedSuffix = m - 1 - j
        
        // Правило плохого символа
        badCharShift = j - badChar[text[t + j]]
        if badCharShift < 1:
            badCharShift = 1
        
        // Правило хорошего суффикса
        goodSuffixShift = goodSuffix[matchedSuffix]
        
        // Выбираем максимальный сдвиг
        t = t + max(badCharShift, goodSuffixShift)
    
    return -1
```

### 5.10 Анализ сложности

- **Препроцессинг:** O(m + σ)
- **Поиск:**
  - Лучший случай: **O(n/m)** — субли­нейная сложность
  - Худший случай: O(n + m) с правилом Галиля
  - Средний случай: O(n)

### 5.11 Правило Галиля (Galil Rule)

В 1979 году **Цви Галиль** предложил оптимизацию, гарантирующую **линейную сложность O(n)** в худшем случае.

**Идея:** После нахождения совпадения (или частичного совпадения) запоминаем границу уже проверенной части текста и не сравниваем её повторно.

## 6. Сравнение алгоритмов

### 6.1 Теоретическое сравнение

| Критерий | Наивный | KMP | BMH | Boyer-Moore |
|----------|---------|-----|-----|-------------|
| Лучший случай | O(n) | O(n) | O(n/m) | O(n/m) |
| Худший случай | O(n·m) | O(n) | O(n·m) | O(n) |
| Средний случай | O(n) | O(n) | O(n) | O(n) |
| Доп. память | O(1) | O(m) | O(σ) | O(m+σ) |
| Направление сравнения | → | → | ← | ← |

### 6.2 Практические рекомендации

**Используйте Boyer-Moore (или BMH), когда:**
- Алфавит большой (например, ASCII, Unicode)
- Образец длинный
- Нужна максимальная скорость на типичных данных
- Текст — естественный язык

**Используйте KMP, когда:**
- Алфавит маленький (например, ДНК: {A, C, G, T})
- Важна гарантированная линейная сложность
- Образец содержит много повторяющихся паттернов

### 6.3 Результаты исследований

Согласно исследованиям, алгоритм Бойера-Мура:
- На 37-40% быстрее KMP на типичных текстовых данных
- Показывает **субли­нейную** производительность на больших алфавитах
- Широко используется в текстовых редакторах (grep, vim, emacs)

## 7. Варианты и модификации

### 7.1 Turbo Boyer-Moore

Комбинирует классический BM с дополнительной оптимизацией: запоминает длину совпавшего суффикса и использует её для пропуска уже проверенных символов.

### 7.2 Apostolico-Giancarlo

Хранит длины совпавших суффиксов для каждой позиции, что позволяет пропускать сравнения. Требует дополнительную память O(n).

### 7.3 Quick Search (Sunday)

Смотрит на символ текста **сразу после** текущего окна образца (позиция `t + m`), что иногда даёт ещё большие сдвиги.

### 7.4 Zhu-Takaoka

Использует два последних символа образца для построения таблицы сдвигов, что эффективнее для некоторых типов данных.

### 8.3 Поиск всех вхождений

```cpp
std::vector<int> findAll(const std::string& text, const std::string& pattern) {
    std::vector<int> positions;
    int pos = 0;
    
    while (pos <= text.length() - pattern.length()) {
        int found = boyerMooreSearch(text.substr(pos), pattern);
        if (found == -1) break;
        
        positions.push_back(pos + found);
        pos += found + 1;  // Продолжаем поиск со следующей позиции
    }
    
    return positions;
}
```

### 8.4 Оптимизация для множественного поиска

Если ищем один и тот же образец в разных текстах, таблицы сдвигов создаём **один раз**:

```cpp
class BoyerMooreSearcher {
    std::string pattern;
    std::array<int, 256> badChar;
    std::vector<int> goodSuffix;
    
public:
    explicit BoyerMooreSearcher(const std::string& pattern);
    int search(const std::string& text) const;
};
```

## 10. Итоги

1. **Наивный алгоритм** — прост, но неэффективен: O(n·m)

2. **Boyer-Moore-Horspool** — использует только правило плохого символа:
   - Прост в реализации
   - O(n/m) в лучшем случае
   - O(n·m) в худшем случае

3. **Boyer-Moore** — использует оба правила (плохой символ + хороший суффикс):
   - Более сложная реализация
   - O(n/m) в лучшем случае
   - O(n) в худшем случае (с правилом Галиля)
   - **Наиболее эффективен на практике**

4. **Ключевая идея:** сравнение справа налево позволяет делать **большие сдвиги**, пропуская множество заведомо неподходящих позиций.
