## 3. Реализованные Компоненты

### 3.1 Интерфейс Хеш-таблицы

```cpp
class IHashTable {
public:
    virtual void insert(int key, int value) = 0;
    virtual bool search(int key, int& value) const = 0;
    virtual bool remove(int key) = 0;
    virtual size_t size() const = 0;
    virtual bool empty() const = 0;
    virtual double loadFactor() const = 0;
    virtual void clear() = 0;
    virtual std::string getName() const = 0;
};
```


### 3.2 Хеш-функции (4 варианта)

| Функция | Формула | Применение |
| :-- | :-- | :-- |
| **Simple** | `key % capacity` | Базовый случай |
| **Multiplicative** | `(key × φ) % capacity` | Golden ratio (0.618...) |
| **Bitwise** | XOR операции со сдвигами | Хорошее распределение |
| **Java-like** | Имитация HashMap Java | Стандартный подход |

**Пример сравнения** (key=12345, capacity=11):

```
Simple:          3
Multiplicative:  6
Bitwise:         4
Java-like:       0
```


### 3.3 Метод Цепочек (Chaining)

**Особенности:**

- ✅ Каждый индекс содержит связный список
- ✅ Load factor может быть > 1.0
- ✅ Простое удаление (физическое)
- ✅ Плохая локальность кэша

**Ключевые методы:**

```cpp
void insert(int key, int value);  // O(1) среднее, O(n) худшее
bool search(int key, int& value);  // O(1) среднее, O(n) худшее
bool remove(int key);              // O(1) среднее, O(n) худшее
void rehash();                      // O(n), при LF > 0.75
```


### 3.4 Открытая Адресация (Open Addressing)

**Особенности:**

- ✅ Все элементы в одном массиве
- ✅ Поддержка 3 методов пробирования
- ✅ Ленивое удаление (DELETED маркеры)
- ✅ Хорошая локальность кэша

**Состояния ячеек:**

```cpp
enum CellState {
    EMPTY,      // Никогда не была занята
    ACTIVE,     // Содержит активный элемент
    DELETED     // Удалена (ленивое удаление)
};
```

**Методы пробирования:**

1. **Линейное:** `h(k,i) = (h'(k) + i) % m`
2. **Квадратичное:** `h(k,i) = (h'(k) + i + i²) % m` ✅
3. **Двойное хеширование:** `h(k,i) = (h'(k) + i×h''(k)) % m`

### 4.2 Пример Успешного Теста

```cpp
TEST(OpenAddressingHashTableTest, QuadraticProbing) {
    OpenAddressingHashTable<SimpleHashFunction<>, SecondaryHashFunction<>> table(7, 0.7);
    
    // Элементы с коллизиями: 0, 7, 14 все хешируются в индекс 0
    table.insert(0, 10);
    table.insert(7, 70);    // Коллизия → квадратичное пробирование
    table.insert(14, 140);  // Коллизия → квадратичное пробирование
    
    EXPECT_EQ(table.size(), 3);
    
    int value = 0;
    EXPECT_TRUE(table.search(0, value));
    EXPECT_EQ(value, 10);
    EXPECT_TRUE(table.search(7, value));
    EXPECT_EQ(value, 70);
    EXPECT_TRUE(table.search(14, value));
    EXPECT_EQ(value, 140);
}
```

## 6. Анализ Производительности

### 6.1 Преимущества Chaining

✅ **Скорость операций**

- Вставка: 24.4 ms для 100K элементов
- Поиск: 24.2 ms для 10K элементов
- Удаление: 24.2 ms для 50K элементов

✅ **Гибкость**

- Load factor может быть > 1.0
- Нет проблем с кластеризацией
- Простое удаление

❌ **Недостатки**

- Плохая локальность кэша (-10-15% на современных CPU)
- Дополнительная память для указателей
- Cache misses при обходе цепочек


### 6.2 Преимущества Open Addressing

✅ **Преимущества**

- Лучшая локальность кэша (в среднем +5-10%)
- Меньше памяти на элемент
- Лучше для SSD-like работы

❌ **Недостатки**

- Медленнее на 7-14% в нашем бенчмарке
- Требует load factor < 1.0
- Проблема кластеризации
- Сложное ленивое удаление


### 6.3 Рекомендации

```
Использовать Chaining если:
  • Операции удаления частые
  • Load factor может меняться
  • Простота кода важнее производительности
  • Много памяти доступно

Использовать Open Addressing если:
  • Операции чтения преобладают
  • Фиксированный размер таблицы
  • Критична локальность кэша
  • Память ограничена
```
