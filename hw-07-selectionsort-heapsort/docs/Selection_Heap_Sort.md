## Selection Sort

### Принцип работы

Selection Sort (сортировка выбором) - один из простейших алгоритмов сортировки. Основная идея:

1. Разделяем массив на две части: отсортированную (слева) и неотсортированную (справа)
2. На каждой итерации находим минимальный элемент в неотсортированной части
3. Меняем его местами с первым элементом неотсортированной части
4. Граница отсортированной части сдвигается на один элемент вправо

### Визуализация

```
Исходный массив: [64, 25, 12, 22, 11]

Итерация 1: Находим минимум (11), меняем с первым
[11 | 25, 12, 22, 64]
 ^    минимум найден

Итерация 2: Находим минимум (12) в оставшейся части
[11, 12 | 25, 22, 64]
     ^    минимум найден

Итерация 3: Находим минимум (22)
[11, 12, 22 | 25, 64]
         ^    минимум найден

Итерация 4: Находим минимум (25)
[11, 12, 22, 25 | 64]
             ^    минимум найден

Результат: [11, 12, 22, 25, 64]
```

### Псевдокод

```
procedureelectionSort(arr)
    n = length(arr)
    for i = 0 to n-2 do
        minIdx = i
        for j = i+1 to n-1 do
            if arr[j] < arr[minIdx] then
                minIdx = j
        if minIdx != i then
            swap(arr[i], arr[minIdx])
```

### Анализ сложности

#### Временная сложность

**Лучший случай**: O(n²)
- Даже если массив уже отсортирован, алгоритм все равно выполняет все сравнения

**Средний случай**: O(n²)
- На i-й итерации выполняется (n-i-1) сравнений
- Общее количество сравнений: (n-1) + (n-2) + ... + 1 = n(n-1)/2 = O(n²)

**Худший случай**: O(n²)
- Сложность не зависит от начального порядка элементов

#### Пространственная сложность

O(1) - алгоритм работает на месте (in-place), не требует дополнительной памяти

#### Количество операций

- **Сравнения**: Всегда n(n-1)/2 ≈ O(n²)
- **Обмены**: Максимум n-1 (минимальное среди алгоритмов O(n²))

### Свойства

| Свойство | Значение |
|----------|----------|
| Стабильность | Нестабильная* |
| Адаптивность | Нет |
| Сортировка на месте | Да |
| Онлайн-алгоритм | Нет |

*Можно сделать стабильной, используя сдвиги вместо обменов

### Преимущества

1. **Простота реализации** - один из самых простых алгоритмов
2. **Минимум обменов** - максимум n-1 обмен (полезно, когда обмен дорогой)
3. **Работает на месте** - O(1) дополнительной памяти
4. **Предсказуемость** - всегда выполняет одинаковое количество сравнений

### Недостатки

1. **Квадратичная сложность** - O(n²) даже в лучшем случае
2. **Не адаптивен** - не использует преимущество частично отсортированных данных
3. **Нестабилен** - не сохраняет относительный порядок равных элементов
4. **Неэффективен** - практически бесполезен для больших массивов

### Применение

**Хорошо подходит для:**
- Небольших массивов (n < 50)
- Когда обмен элементов - дорогая операция
- Обучения основам алгоритмов сортировки

**Не подходит для:**
- Больших массивов
- Когда важна производительность
- Когда нужна стабильность

## Heap Sort

### Принцип работы

Heap Sort (пирамидальная сортировка) использует структуру данных "двоичная куча" (binary heap).

**Куча (Heap)** - это полное двоичное дерево, где:
- **Max-heap**: значение каждого узла ≥ значений его потомков
- **Min-heap**: значение каждого узла ≤ значений его потомков

Для сортировки по возрастанию используем max-heap.

### Алгоритм

1. **Построение кучи** (Build Max-Heap)
   - Превращаем массив в max-heap
   - Время: O(n)

2. **Извлечение элементов** (Extract Max)
   - Корень (максимальный элемент) меняем с последним
   - Уменьшаем размер кучи
   - Восстанавливаем свойство кучи (heapify)
   - Повторяем n-1 раз
   - Время: O(n log n)

### Представление кучи в массиве

Для узла с индексом i:
- **Левый потомок**: 2*i + 1
- **Правый потомок**: 2*i + 2
- **Родитель**: (i-1) / 2

```
Массив: [90, 85, 70, 60, 80, 30, 50]

Представление в виде дерева:
        90
       /  \
     85    70
    / \   / \
   60 80 30 50
```

### Визуализация сортировки

```
Исходный массив: [4, 10, 3, 5, 1]

Шаг 1: Построение max-heap
[10, 5, 3, 4, 1]
Дерево:     10
           /  \
          5    3
         / \
        4   1

Шаг 2: Извлечение максимума (10)
Меняем 10 и 1: [1, 5, 3, 4 | 10]
Heapify:       [5, 4, 3, 1 | 10]

Шаг 3: Извлечение максимума (5)
Меняем 5 и 1:  [1, 4, 3 | 5, 10]
Heapify:       [4, 1, 3 | 5, 10]

Шаг 4: Извлечение максимума (4)
Меняем 4 и 3:  [3, 1 | 4, 5, 10]
Heapify:       [3, 1 | 4, 5, 10]

Шаг 5: Извлечение максимума (3)
Меняем 3 и 1:  [1 | 3, 4, 5, 10]

Результат: [1, 3, 4, 5, 10]
```

### Псевдокод

```
procedure heapSort(arr)
    buildMaxHeap(arr)
    for i = length(arr)-1 downto 1 do
        swap(arr[0], arr[i])
        heapify(arr, i, 0)

procedure buildMaxHeap(arr)
    n = length(arr)
    for i = n/2 - 1 downto 0 do
        heapify(arr, n, i)

procedure heapify(arr, n, i)
    largest = i
    left = 2*i + 1
    right = 2*i + 2
    
    if left < n and arr[left] > arr[largest] then
        largest = left
    if right < n and arr[right] > arr[largest] then
        largest = right
    
    if largest != i then
        swap(arr[i], arr[largest])
        heapify(arr, n, largest)
```

### Анализ сложности

#### Временная сложность

**Лучший случай**: O(n log n)

**Средний случай**: O(n log n)

**Худший случай**: O(n log n)

**Детальный анализ:**
- Построение кучи: O(n)
- n-1 извлечений, каждое с heapify: O(log n)
- Итого: O(n) + O(n log n) = O(n log n)

#### Пространственная сложность

O(1) - рекурсия heapify требует O(log n) стека, но это не учитывается

### Свойства

| Свойство | Значение |
|----------|----------|
| Стабильность | Нестабильная |
| Адаптивность | Нет |
| Сортировка на месте | Да |
| Онлайн-алгоритм | Нет |
| Гарантии | Всегда O(n log n) |

### Преимущества

1. **Гарантированная производительность** - O(n log n) всегда
2. **Работает на месте** - O(1) дополнительной памяти
3. **Нет худшего случая** - в отличие от QuickSort
4. **Предсказуемость** - время работы не зависит от входных данных

### Недостатки

1. **Нестабильность** - не сохраняет порядок равных элементов
2. **Плохая локальность** - прыжки по памяти снижают производительность кэша
3. **Практически медленнее QuickSort** - несмотря на одинаковую асимптотику
4. **Сложность реализации** - сложнее простых алгоритмов O(n²)

### Применение

**Хорошо подходит для:**
- Больших массивов
- Когда нужна гарантия O(n log n)
- Системы реального времени (предсказуемость)
- Когда критична память

**Не подходит для:**
- Когда нужна стабильность
- Маленькие массивы (overhead)
- Когда QuickSort доступен и приемлем

## Сравнение алгоритмов

### Таблица характеристик

| Характеристика | Selection Sort | Heap Sort |
|----------------|----------------|-----------|
| **Временная сложность** |
| Лучший случай | O(n²) | O(n log n) |
| Средний случай | O(n²) | O(n log n) |
| Худший случай | O(n²) | O(n log n) |
| **Пространственная сложность** | O(1) | O(1) |
| **Стабильность** | Нет | Нет |
| **Адаптивность** | Нет | Нет |
| **Сравнения** | ~n²/2 | ~n log n |
| **Обмены** | ≤ n | ~n log n |
| **Сортировка на месте** | Да | Да |
| **Сложность кода** | Простая | Средняя |

### Производительность на практике

**Selection Sort:**
- n = 100: ~0.1 мс
- n = 1,000: ~10 мс
- n = 10,000: ~1 с
- n = 100,000: ~100 с (непрактично)

**Heap Sort:**
- n = 100: ~0.05 мс
- n = 1,000: ~0.5 мс
- n = 10,000: ~7 мс
- n = 100,000: ~90 мс
- n = 1,000,000: ~1 с

### Рекомендации по выбору

**Используйте Selection Sort если:**
- Массив очень маленький (n < 50)
- Обмен элементов очень дорогой
- Нужна простота реализации
- Это учебная задача

**Используйте Heap Sort если:**
- Нужна гарантия O(n log n)
- Критична память (O(1))
- Неприемлем худший случай QuickSort
- Массив достаточно большой (n > 1000)

**В большинстве случаев:**
- Для общего назначения: **QuickSort** или **IntroSort**
- Для стабильности: **Merge Sort** или **Timsort**
- Для почти отсортированных: **Insertion Sort**

## Интересные факты

### Selection Sort
- Минимальное количество обменов среди всех алгоритмов сортировки на основе сравнений
- Используется в некоторых embedded системах из-за простоты
- Бесполезен на практике для n > 100

### Heap Sort
- Изобретен Дж. Уильямсом в 1964 году
- Используется в priority queue
- Основа алгоритма HeapSelect для поиска k-го элемента
- IntroSort переключается на HeapSort при плохом QuickSort
