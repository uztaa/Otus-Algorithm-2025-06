## Реализованные алгоритмы

### 1. RLECompressor — базовый Run-Length Encoding (2+2+2 балла)

**Формат:** каждая серия кодируется парой `(count, byte)`
- `count` ∈ [1, 255] — количество повторов
- `byte` — повторяющийся байт

**Пример:**
```
Вход:  AAABBC (6 байт)
Выход: 03 41 02 42 01 43 (6 байт)
```

**Недостаток:** данные без повторов раздуваются в 2 раза!

### 2. PackBitsCompressor — улучшенный RLE (2 балла)

Стандартный алгоритм из TIFF/MacPaint.

**Формат:** флаговый байт N определяет режим:
- N ∈ [0, 127]: следующие (N+1) байт — литералы
- N ∈ [-127, -1]: следующий байт повторяется (1-N) раз
- N = -128: NOP

**Пример:**
```
Вход:  AAABCDEF
Выход: FE 41 04 42 43 44 45 46
       ↑     ↑
       -2=3× 4=5 литералов
```

**Преимущества:**
- Не раздувает данные без повторов
- Worst-case overhead: ~1% (1 флаг на 128 байт)

## Методология тестирования

Для сравнения работы алгоритмов на разных типах данных используется **генерация тестовых данных**, имитирующих реальные файлы:

| Тип данных | Генератор | Имитирует |
|------------|-----------|-----------|
| **Repeating** | Длинные серии одинаковых байт | Иконки, чёрно-белые изображения |
| **Pattern** | Паттерн 0,0,0,1,1,1,2,2,2... | Градиенты, простая графика |
| **Text** | Lorem Ipsum | Текстовые файлы, исходный код |
| **Random** | Случайные байты | Фотографии, аудио, видео |
| **Compressed** | Случайные с высокой энтропией | ZIP-архивы, уже сжатые файлы |

Этот подход обеспечивает:
- **Воспроизводимость** — одинаковые результаты при повторных запусках
- **Контроль размера** — все датасеты одинакового размера (100 KB)
- **Покрытие сценариев** — от идеального до худшего случая для RLE

`Я проверил на реальных типах данных (txt, zip, mp4): коэффициенты сжатия очень близки к результатам бенчмака.`

## Описание тестовых сценариев

| Тест | Характеристика | Ожидаемый результат RLE | Ожидаемый результат PackBits |
|------|----------------|-------------------------|------------------------------|
| **Repeating** | Длинные серии | Отличное сжатие (10-50x) | Отличное сжатие (10-50x) |
| **Pattern** | Короткие серии по 3 | Хорошее сжатие (1.5x) | Хорошее сжатие (1.5x) |
| **Text** | Мало повторов | Раздувание (~2x) | Минимальный overhead (<1%) |
| **Random** | Нет повторов | Раздувание (2x) | Минимальный overhead (<1%) |
| **Compressed** | Высокая энтропия | Раздувание (2x) | Минимальный overhead (<1%) |

## Выводы

1. **Базовый RLE** подходит только для данных с гарантированными длинными сериями повторов (чёрно-белые изображения, иконки). На произвольных данных размер удваивается.

2. **PackBits** решает проблему раздувания за счёт режима литералов. На данных без повторов overhead составляет менее 1%.

3. **Рекомендации по выбору:**
   - Простые изображения с большими однотонными областями → RLE или PackBits
   - Универсальное сжатие → DEFLATE (zlib)
   - Максимальное сжатие → LZMA (7-zip)

4. **Ограничения RLE-семейства:**
   - Не учитывает статистику символов (в отличие от Huffman)
   - Не использует словарь (в отличие от LZ77)
   - Эффективен только для специфических типов данных

## API использования

```cpp
#include "compression/PackBitsCompressor.hpp"

PackBitsCompressor compressor;

// Сжатие
std::vector<uint8_t> original = {0x41, 0x41, 0x41, 0x42, 0x43};
auto compressed = compressor.compress(original);

// Распаковка
auto decompressed = compressor.decompress(compressed);

// Проверка
assert(original == decompressed);

// Метрики
double ratio = ICompressor::compressionRatio(original.size(), compressed.size());
double percent = ICompressor::compressionPercent(original.size(), compressed.size());
```
